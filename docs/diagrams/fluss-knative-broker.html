Fluss as Knative Broker — Event Flow

CloudEvent lifecycle from producer to subscriber through Fluss Log Tables

Fluss IS the Broker

  • • Each namespace → Fluss database
  • • Each broker → Fluss log table
  • • Arrow IPC columnar format on disk
  • • Adapted Kafka replication protocol
  • • Sub-second write latency via RocksDB

Independent Trigger Cursors

  • • Each trigger has its own scan cursor
  • • Triggers don't compete for resources
  • • Filter by CloudEvent attributes
  • • Credit-based backpressure per lane
  • • At-least-once delivery with retry

Failure Isolation

  • • Slow subscriber → pause lane only
  • • Poison message → DLQ after retries
  • • Iceberg failure → broker continues
  • • Per-trigger backpressure (no global stall)
  • • Failed events replayable from DLQ
